# 仕様書品質チェック

指定された仕様書ファイル（$ARGUMENTS）に対して、以下の観点で品質チェックを行う。
引数が無い場合は `docs/spec_*.md`（テンプレートを除く）全件を対象にする。

# チェック観点

## 1. テンプレート準拠チェック

`docs/spec_x-x_specname_template.md` の構成と比較し、必須セクションが揃っているか確認する。

- [ ] `# 仕様書: ＜機能番号＞＜機能名＞` が記載されている
- [ ] `## 1. 概要` が存在し、内容が記載されている
- [ ] `## 2.` 関連資料セクションが存在する
- [ ] `## 3. 機能詳細` が存在する
- [ ] `### 3.1 画面要素` が存在する
- [ ] `### 3.2 処理ロジック` が存在する
- [ ] `## 4. 考慮事項・制限事項` が存在し、内容が記載されている
- [ ] `## 5. テスト項目` が存在し、テスト項目が1件以上ある

## 2. spec_list.md との整合性

`docs/spec_list.md` を参照し、以下を確認する。

- [ ] 仕様書の機能番号が spec_list.md に存在する
- [ ] 仕様書の機能名が spec_list.md の記載と一致する
- [ ] 仕様書のファイル名が `spec_{番号}_{英語名}.md` の命名規則に従っている

## 3. 内容の品質

- [ ] 概要がその機能だけで理解できる説明になっている
- [ ] 画面要素に具体的な UI 要素やレイアウト情報が含まれている
- [ ] 処理ロジックに具体的なデータ構造やアルゴリズムが記載されている
- [ ] 他機能との境界が明確である（考慮事項・制限事項で言及されている）
- [ ] 参照先（他の仕様書、アセットファイル等）のパスが正しく存在する

## 4. テスト項目の品質

- [ ] テスト項目がカテゴリ分けされている
- [ ] 各テスト項目が具体的で、合否を判定できる記述になっている
- [ ] テスト項目が Playwright で自動テスト可能な内容になっている
- [ ] 仕様書に記載された全ての機能に対応するテスト項目がある
- [ ] 正常系だけでなく異常系・境界値のテストも含まれている

## 5. 仕様の曖昧さチェック

以下のような曖昧な記述が無いか確認する。

- 「など」「等」で終わる記述に具体的な列挙がないもの
- 数値が未定義のもの（サイズ、タイミング、閾値）
- 「適切に」「必要に応じて」など判断基準が不明な表現
- 主語が不明確な文

# 出力形式

チェック結果を以下の形式で報告する。

```
## 品質チェック結果: {ファイル名}

### 合格項目
- ✅ {チェック項目}: {簡潔な説明}

### 指摘事項
- ⚠️ {チェック項目}: {問題の説明と改善案}

### 総合評価
{全体的な評価コメント}
```

問題が見つかった場合は改善案を具体的に提示し、ユーザーの承認を得てから修正すること。
