# 仕様書ルール
1.以下の項目を持つこと
  - 基本情報（概要・対応デバイス）
  - 画面設計（モック資料リンク・画面項目定義）
  - 処理ロジック
  - データ設計（データ元、データ構造、API設計）
  - 非機能要件（パフォーマンス要件）
  - テスト方針

2.概要は分かりやすい文章で、誰が何のために使う機能かを書くこと
3.画面が必要な機能ではモックアップ資料が必ず存在すること

画面を持つ機能はモックアップを必ず用意する
の有無を確認し、なければ他の結果に関わらず ❌ とし冒頭で指摘する
- ステップ1「何を作るか」: 概要、位置づけ（spec_list.md 整合）、必要な技術、参照ドキュメント（実在確認）
- ステップ2「どのように作るか」: データ構造（型・フォーマット・サンプル値）、処理ロジック（条件分岐・曖昧表現なし）、画面構成（サイズ・色・位置の数値）
- ステップ3「どう品質を担保するか」: テスト方針（方向性があれば合格、個別ケース網羅は不要）
- 明確性チェック: 曖昧表現なし、主語目的語明確、数値定義あり、用語統一
- 「何を作るか」が不明確なら他が詳細でも要修正
- 「どのように作るか」は別の開発者がそのまま実装着手できるレベル
- 画面機能にモックアップなければ他が完璧でも要修正
- 曖昧さは最大の敵。解釈の余地がある記述は指摘する
- 出力は指定フォーマットに従う（ステップ0判定→サマリー→詳細チェック→改善項目→Issue化準備状況）
